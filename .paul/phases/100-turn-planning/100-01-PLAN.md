<objective>
## Goal
Add multi-ordering turn planning: try several unit orderings per turn, evaluate the final state of each, and pick the ordering that produces the best board state. This enables implicit coordination — units that move first affect later units' decisions.

## Purpose
Phase 99 gave each unit a 1-ply lookahead, but units still act in arbitrary HashMap iteration order. Order matters: if unit A kills enemy X first, unit B can focus on enemy Y instead of wasting damage on X. Trying multiple orderings finds naturally coordinated plans.

## Output
Modified `ai_plan_turn` and `ai_take_turn` in `norrust_core/src/ai.rs` using a `plan_full_turn()` helper that runs K orderings and picks the best.
</objective>

<task type="auto">
  <name>Task 1: Extract plan_full_turn helper with multi-ordering</name>
  <files>norrust_core/src/ai.rs</files>
  <action>
    1. Add `fn plan_full_turn(state: &GameState, faction: u8, cheapest_recruit_cost: u32) -> (Vec<ActionRecord>, f32)` that encapsulates the complete turn planning loop and returns (actions, final_score).

    2. Inside `plan_full_turn`:
       a. Compute leader discipline decisions (stay, return_keep) once
       b. Collect non-leader unit IDs into a Vec
       c. For K orderings (K=5 or configurable constant):
          - Clone the state
          - Shuffle unit IDs using a simple deterministic shuffle (e.g., rotate by i, or use state.rng if available)
          - Process leader first (leader discipline), then shuffled non-leader units
          - For each unit: call `plan_unit_action` on the clone, apply the chosen action to the clone, record actions
          - After all units processed: evaluate the final clone state
          - Track best (actions, score)
       d. Also include the original (unshuffled) ordering as one of the K attempts
       e. Return the best (actions, score)

    3. For shuffling: use a simple rotation — ordering `i` starts at index `i % len`. This is deterministic and avoids needing RNG. First ordering = original order. Subsequent orderings rotate the unit list.

    4. Update `ai_plan_turn` to call `plan_full_turn` and return the best actions.

    5. Update `ai_take_turn` to call `plan_full_turn`, then replay the best actions on the real state:
       - For each ActionRecord in the result, apply the corresponding Action to real state
       - Then EndTurn

    6. Performance note: K=5 orderings means ~5x more cloning. In debug mode, balance tests will be slower (~500s). This is acceptable — debug mode is for correctness, not speed. Consider adding a `const TURN_PLAN_ORDERINGS: usize = 5;` at the top of the file for easy tuning.

    Note: Leader always acts first regardless of ordering — only non-leader units are reordered. This preserves leader discipline.
