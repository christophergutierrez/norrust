<objective>
## Goal
Link existing scenarios into a campaign sequence with unit and gold carry-over between scenarios. Player completes scenario 1, surviving units carry to scenario 2 with XP/level/HP preserved. Gold carries over with configurable percentage penalty. Early finish bonus gold rewards efficient play.

## Purpose
This is the final phase of v1.3 Campaign Mode. Phases 28-29 built two standalone scenarios (crossing + ambush); this phase chains them into a multi-scenario campaign experience — the core progression loop that transforms individual battles into a coherent story.

## Output
- `campaigns/tutorial.toml` — Campaign definition file (scenario sequence, carry-over rules)
- `norrust_core/src/campaign.rs` — Campaign logic (TOML schema, veteran serialization, gold carry-over)
- FFI functions for campaign management (load campaign, get survivors, get carry-over gold)
- Love2D campaign flow (campaign mode in scenario selection, auto-progression on victory)
- Integration tests for campaign carry-over logic
</objective>

    4. No changes to schema.rs needed — CampaignDef is independent of existing schemas

    Avoid: Adding campaign state to GameState — campaign progression is managed by the client (Love2D), not the engine. The engine just provides data extraction (survivors, gold).

<task type="auto">
  <name>Task 2: FFI functions for campaign management + veteran placement</name>
  <files>norrust_core/src/ffi.rs, norrust_love/norrust.lua</files>
  <action>
    1. Add FFI functions to `ffi.rs`:
       - `norrust_load_campaign(engine, path) -> *mut c_char`: loads campaign TOML, returns JSON string with campaign metadata (id, name, scenario count, gold_carry_percent, early_finish_bonus, scenario list)
       - `norrust_get_survivors_json(engine, faction) -> *mut c_char`: returns JSON array of VeteranUnit structs for surviving units of given faction
       - `norrust_get_carry_gold(engine, faction, gold_carry_percent, early_finish_bonus) -> i32`: returns calculated carry-over gold for faction
       - `norrust_place_veteran_unit(engine, unit_id, def_id, faction, col, row, hp, xp, xp_needed, advancement_pending) -> i32`: like place_unit_at but sets hp/xp/xp_needed/advancement_pending to carried values instead of registry defaults. Returns 0 on success, -1 on error.

    2. `norrust_place_veteran_unit` implementation:
       - Call `unit_from_registry()` to get full unit with registry stats (attacks, defense, movement_costs, resistances, etc.)
       - Then override: unit.hp = hp, unit.xp = xp, unit.xp_needed = xp_needed, unit.advancement_pending = (advancement_pending != 0)
       - Place via state.place_unit()
       - This preserves registry-derived combat stats while carrying over progression state

    3. Add cdef declarations to `norrust_love/norrust.lua`:
       ```
       const char* norrust_load_campaign(NorRustEngine*, const char*);
       const char* norrust_get_survivors_json(NorRustEngine*, int);
       int norrust_get_carry_gold(NorRustEngine*, int, int, int);
       int norrust_place_veteran_unit(NorRustEngine*, int, const char*, int, int, int, int, int, int, int);
       ```

    4. Add Lua wrapper functions in norrust.lua for each new FFI function

    Avoid: Storing campaign state inside NorRustEngine — the engine is per-scenario. Campaign progression is client-side.
