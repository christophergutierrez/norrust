<objective>
## Goal
Add persistent unit identity via Lua-generated UUIDs and a campaign roster that tracks all player units across scenarios — not just survivors.

## Purpose
Currently, campaign carry-over only remembers surviving units as anonymous VeteranUnit tables. There's no way to track a specific unit's history across scenarios, distinguish recruited units from preset ones, or know which units died. A persistent roster with UUIDs enables unit identity, future features like naming/leveling history, and cleaner save/load.

## Output
- roster.lua module: UUID generation, roster CRUD, engine_id↔UUID mapping
- Campaign flow wired through roster (recruit, carry-over, placement)
- Save/load includes [[roster]] section
</objective>

<task type="auto">
  <name>Task 1: Create roster.lua module</name>
  <files>norrust_love/roster.lua</files>
  <action>
    Create roster.lua with:

    1. `roster.generate_uuid()` — returns a random 8-char hex string using math.random.
       Format: "xxxxxxxx" where x is [0-9a-f]. Seed with os.time() + os.clock() on first call.

    2. `roster.new()` — returns a new roster table: {entries={}, id_map={}}
       - entries: UUID-keyed table of {uuid, def_id, hp, max_hp, xp, xp_needed, advancement_pending, status}
         status is "alive" or "dead"
       - id_map: engine_id (number) → UUID mapping for current scenario

    3. `roster.add(r, def_id, engine_id, hp, max_hp, xp, xp_needed, advancement_pending)` — creates entry with fresh UUID, maps engine_id, returns UUID

    4. `roster.map_id(r, engine_id, uuid)` — maps engine_id → UUID in id_map

    5. `roster.sync_from_engine(r, state)` — given engine state JSON, updates all mapped entries:
       - For each unit in state.units with a mapped UUID: update hp, xp, xp_needed, advancement_pending
       - For mapped UUIDs whose engine_id no longer exists in state.units: set status="dead"

    6. `roster.get_living(r)` — returns array of entries with status=="alive"

    7. `roster.clear_id_map(r)` — clears id_map (called between scenarios)

    8. `roster.to_save_array(r)` — returns array of entry tables for TOML serialization

    9. `roster.from_save_array(arr)` — returns a new roster populated from saved array
  </action>
  <verify>luajit -bl norrust_love/roster.lua /dev/null succeeds</verify>
  <done>AC-1 partial: UUID generation and roster data structure exist</done>
</task>

<task type="auto">
  <name>Task 2: Wire roster into campaign flow</name>
  <files>norrust_love/main.lua, norrust_love/campaign_client.lua</files>
  <action>
    In main.lua:
    1. `require("roster")` at top
    2. Add `campaign_roster = nil` to campaign state vars (nil when no campaign)
    3. When campaign starts (campaign_active set to true in 'c' handler):
       `campaign_roster = roster.new()`
    4. When campaign ends (campaign_active set to false): `campaign_roster = nil`

    In the recruit handler ('r' key, after place_unit succeeds):
    - If campaign_active and campaign_roster:
      `roster.add(campaign_roster, recruit_def_id, placed_unit_id, hp, max_hp, xp, xp_needed, false)`

    After preset units are loaded (in campaign_client.load_campaign_scenario):
    - If first scenario (campaign_index == 0): populate roster from all faction 0 preset units
      by reading state, filtering faction==0, and calling roster.add for each

    On victory (game_over handler, player won):
    - Before carry-over: `roster.sync_from_engine(campaign_roster, norrust.get_state(engine))`
    - Replace `campaign_veterans = survivors` with: veterans derived from roster.get_living()
    - Build campaign_veterans array from living roster entries (same VeteranUnit shape)

    In campaign_client.place_veterans:
    - After placing each veteran via place_veteran_unit, call roster.map_id to link new engine_id to existing UUID
    - Call roster.clear_id_map before placing (new scenario = new engine IDs)

    On F9 load with campaign context:
    - If data.roster exists, rebuild campaign_roster from roster.from_save_array(data.roster)
    - After units placed, re-map engine IDs to roster UUIDs by matching def_id + position
  </action>
  <verify>luajit -bl norrust_love/main.lua /dev/null; luajit -bl norrust_love/campaign_client.lua /dev/null</verify>
  <done>AC-1, AC-2, AC-4, AC-5 satisfied: roster wired through full campaign lifecycle</done>
</task>
