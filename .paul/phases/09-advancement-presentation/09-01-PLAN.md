<objective>
## Goal
Surface XP and advancement state in the Redot presentation layer: XP progress text under each unit
circle, a gold ring visual indicator when `advancement_pending = true`, and an 'A' key interaction
to trigger `apply_advance()` for the selected unit.

## Purpose
Phase 8 delivered the full headless advancement pipeline. Phase 9 closes the loop for human players —
they need to see XP accruing, know when a unit is ready to advance, and have a clear action to
trigger it. Without this, advancement exists but is invisible to the player.

## Output
Modified `norrust_client/scripts/game.gd` with three additions:
- XP progress text drawn below HP on every unit circle
- Gold polyline ring drawn around units with `advancement_pending = true`
- 'A' key handler in `_input()` that calls `_core.apply_advance()` on the selected unit
</objective>

<task type="auto">
  <name>Task 2: Add 'A' key handler for advancement in _input()</name>
  <files>norrust_client/scripts/game.gd</files>
  <action>
    In `_input()`, inside the `if event is InputEventKey:` block, add an `elif` branch for KEY_A
    BEFORE the existing `return` statement:

    ```gdscript
    elif key_event.pressed and not key_event.echo and key_event.keycode == KEY_A:
        if _selected_unit_id != -1:
            var state = _parse_state()
            var active = _core.get_active_faction()
            for unit in state.get("units", []):
                if unit["id"] == _selected_unit_id and unit["faction"] == active \
                        and unit["advancement_pending"]:
                    var result = _core.apply_advance(_selected_unit_id)
                    print("Advance unit %d: code %d" % [_selected_unit_id, result])
                    _clear_selection()
                    queue_redraw()
                    break
    ```

    Placement: add the elif after the KEY_E block, before `return`. The `return` already
    exits after any key event — this new branch runs only when 'A' is pressed.

    Constraints:
    - Only advances if: unit is selected + belongs to active faction + advancement_pending = true
    - Calls `_core.apply_advance(int)` — unit_id is already an int from `_selected_unit_id`
    - Calls `_clear_selection()` + `queue_redraw()` on success (same pattern as move/attack)
    - Prints result code for debugging (same pattern as end_turn, move, attack handlers)
    - Does NOT call `_check_game_over()` — advancement cannot cause a win condition
  </action>
