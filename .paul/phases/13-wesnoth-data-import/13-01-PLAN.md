`/home/chris/git_home/wesnoth/data/core/units/`, expands movement type templates

    Create `tools/scrape_wesnoth.py`. The scraper is a self-contained Python 3 script
    (~150 lines) using only the stdlib. Create `tools/` directory first.

    ## Step 1: Parse movement types from units.cfg

    Source: `/home/chris/git_home/wesnoth/data/core/units.cfg`

    Parse all `[movetype]` ... `[/movetype]` blocks. Within each:
    - Extract `name=` as the movetype key
    - Parse `[movement_costs]` ... `[/movement_costs]` as `{terrain: int}` dict
    - Parse `[defense]` ... `[/defense]` as `{terrain: int}` dict
    - Parse `[resistance]` ... `[/resistance]` as `{attack_type: int}` dict

    Build: `MOVETYPES: dict[str, dict]` with keys `movement_costs`, `defense`, `resistance`.

    **Resistance conversion**: Wesnoth resistance values are "% of damage taken"
    (100 = normal, 90 = 10% less damage = resistant, 120 = 20% more damage = weak).
    Our Rust combat formula is `damage * (100 + our_value) / 100`.
    Conversion: `our_value = wesnoth_value - 100`
    - Wesnoth 100 → our 0 (normal, no modifier)
    - Wesnoth 90 → our -10 (resistant, takes 10% less)
    - Wesnoth 120 → our +20 (weak, takes 20% more)

    ## Step 2: Parse all unit_type blocks

    Source: All `*.cfg` files under `/home/chris/git_home/wesnoth/data/core/units/`
    (recursive glob).

    Parse `[unit_type]` ... `[/unit_type]` blocks. Within each, extract:

    | WML field | TOML field | Notes |
    |-----------|------------|-------|
    | `id=` | `id` | Preserve original casing |
    | `name= _ "..."` | `name` | Strip `_ "` prefix and `"` suffix |
    | `race=` | `race` | |
    | `hitpoints=` | `max_hp` | int |
    | `movement=` | `movement` | int |
    | `experience=` | `experience` | int |
    | `level=` | `level` | int, default 1 |
    | `alignment=` | `alignment` | |
    | `advances_to=` | `advances_to` | Comma-split, strip whitespace, filter empty |
    | `cost=` | `cost` | int, default 0 |
    | `usage=` | `usage` | strip surrounding whitespace |
    | `abilities_list=` | `abilities` | Space-split, filter empty |
    | `movement_type=` | (lookup key) | Look up in MOVETYPES dict |

    For `[attack]` ... `[/attack]` blocks (may be multiple), extract:
    - `name=` → `id`
    - `description= _ "..."` → `name` (strip `_ "` and `"`)
    - `damage=` → `damage` (int)
    - `number=` → `strikes` (int)
    - `type=` → `attack_type`
    - `range=` → `range`

    For `[resistance]` blocks (unit-level overrides), parse as `{attack_type: int}`.
    Apply override: merge over movetype resistances, converting with `wesnoth - 100`.

    **Units to skip (do not generate TOML):**
    - `movement_type` value contains `{` (macro placeholder, e.g. `{MOVETYPE_ID}`)
    - `movement_type` is empty or absent
    - No `[attack]` blocks (non-combat units like ships)
    - `movement_type` not found in MOVETYPES dict

    ## Step 3: Write unit TOMLs

    For each accepted unit, write `data/units/{safe_id}.toml`.

    **safe_id**: `id.lower().replace(' ', '_').replace("'", '').replace(',', '').replace('-', '_')`

    Do NOT overwrite files that already exist (skip if `os.path.exists(outpath)`).
    This preserves our 4 custom unit TOMLs (fighter.toml etc.).

    TOML format (see fighter.toml for reference):
    ```toml
    id = "Spearman"
    name = "Spearman"
    race = "human"
    level = 1
    experience = 40
    max_hp = 33
    movement = 5
    alignment = "lawful"
    cost = 12
    usage = "fighter"
    abilities = []
    advances_to = ["Swordsman"]

    [[attacks]]
    id = "Spear"
    name = "Spear"
    damage = 7
    strikes = 3
    attack_type = "pierce"
    range = "melee"
    specials = []

    [resistances]
    blade = 0
    pierce = 0
    impact = 0
    fire = 0
    cold = 0
    arcane = -10

    [movement_costs]
    flat = 1
    hills = 2
    mountains = 3
    forest = 2
    village = 1
    castle = 1
    cave = 2
    frozen = 3
    fungus = 2
    sand = 2
    shallow_water = 3
    reef = 2
    swamp_water = 3

    [defense]
    flat = 60
    hills = 50
    mountains = 40
    forest = 50
    village = 40
    castle = 40
    cave = 60
    frozen = 80
    fungus = 50
    sand = 70
    shallow_water = 80
    reef = 70
    swamp_water = 80
    ```

    Use Python's `tomllib` (stdlib in 3.11+) is read-only; write manually using
    string formatting, NOT a TOML library. This keeps the scraper dependency-free.

    Write the TOML file using f-strings. For `advances_to` list, write:
    `advances_to = ["{item1}", "{item2}"]` or `advances_to = []`.

    For `abilities`, write: `abilities = ["ability1"]` or `abilities = []`.

    For `[[attacks]]`, write one `[[attacks]]` block per attack in order.
    For `specials`, always write `specials = []` (we don't parse attack specials yet).

    ## Step 4: Write terrain TOMLs

    Collect all terrain codes found in `[movement_costs]` across all movetypes.
    For each terrain code not already having a file in `data/terrain/`:

    Write `data/terrain/{terrain_id}.toml`:
    ```toml
    id = "{terrain_id}"
    name = "{terrain_id.replace('_', ' ').title()}"
    symbol = "{terrain_id[0]}"
    default_defense = {defense_value_from_smallfoot}
    default_movement_cost = {cost_value_from_smallfoot}
    healing = 0
    ```

    Use `smallfoot` defense/cost as the canonical defaults. Skip existing files.

    ```
    Terrain files written: N
    ```

    ```rust
    fn test_wesnoth_units_load() {

            .parent()
            .expect("norrust_core has a parent dir")

            .expect("all unit TOMLs must load without error");

        assert_eq!(spearman.level, 1, "Spearman level");
        assert_eq!(spearman.alignment, "lawful", "Spearman alignment");

        );
    ```
