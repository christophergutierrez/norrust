<objective>
## Goal
Port `norrust_client/scripts/game.gd` (706 lines) to a Love2D client with full feature parity. The Love2D client uses LuaJIT FFI to call the C ABI bridge from Phase 25, producing a fully playable hex strategy game.

## Purpose
Completes the Love2D migration — once this works, Redot can be removed (Phase 27). Love2D is a better architectural fit: the game only uses procedural 2D drawing, not Godot's scene tree or physics.

## Output
- `norrust_love/conf.lua` — Love2D window configuration
- `norrust_love/norrust.lua` — LuaJIT FFI bindings module wrapping all 36 C ABI functions
- `norrust_love/main.lua` — Complete game client (hex rendering, input, HUD, panels, camera, AI)
</objective>

    **norrust.lua — LuaJIT FFI wrapper module:**

    Use `local ffi = require("ffi")` and `local json = require("norrust_love.json")` (or inline JSON parser — see below).

    1. `ffi.cdef` block declaring all 36 C function signatures from ffi.rs:
       - `NorRustEngine* norrust_new()`
       - `void norrust_free(NorRustEngine* engine)`
       - `void norrust_free_string(char* s)`
       - `void norrust_free_int_array(int* arr, int len)`
       - All 32 bridge functions with exact signatures matching ffi.rs

    2. Load the shared library:
       - `ffi.load()` with path to `libnorrust_core.so`
       - Path resolution: look for `../norrust_core/target/debug/libnorrust_core.so` relative to the love project, OR accept a `NORRUST_LIB` environment variable

    3. Lua-friendly wrapper functions in returned module table:
       - `M.new()` — returns engine pointer (use `ffi.gc` to attach norrust_free as destructor)
       - String return helpers: call C function, convert via `ffi.string()`, free via `norrust_free_string()`, return Lua string
       - Int array helpers: call `norrust_get_reachable_hexes()`, read pairs, free array, return Lua table of {col, row} pairs
       - All functions should accept/return Lua-native types (strings, numbers, tables) — no raw C pointers exposed to main.lua
       - `M.get_state(engine)` — calls get_state_json, parses JSON, returns Lua table (use a minimal JSON decoder)

    **JSON Parsing:**
    Love2D / LuaJIT does not include a JSON parser. Include a minimal inline JSON decoder (< 100 lines, pure Lua) inside norrust.lua or as a separate tiny module. Only needs to handle the StateSnapshot format: objects, arrays, strings, numbers, booleans, null. No need for encoding.

    Do NOT use external Lua dependencies — everything must be self-contained.

<task type="auto">
  <name>Task 2: Create main.lua — complete game port from game.gd</name>
  <files>norrust_love/main.lua</files>
  <action>
    Port `norrust_client/scripts/game.gd` to Love2D `main.lua`. The game.gd IS the specification — every constant, state variable, function, and drawing command has a direct Love2D equivalent. Use `local norrust = require("norrust")` for all Rust calls.

    **Structure (mirrors game.gd sections):**

    1. **Constants** — Direct copy from game.gd:
       - BOARD_COLS=8, BOARD_ROWS=5
       - HEX_RADIUS=96, HEX_CELL_W=166, HEX_CELL_H=192
       - COLOR_FLAT={0.29, 0.49, 0.31}
       - CAMERA_PAN_SPEED=500, CAMERA_LERP_SPEED=8
       - RH_STRIDE=2, RH_COL=1, RH_ROW=2 (1-indexed in Lua)
       - Faction colors: blue={0.25, 0.42, 0.88}, red={0.80, 0.12, 0.12}

    2. **State variables** — Same as game.gd, using Lua tables:
       - engine, selected_unit_id, reachable_cells, game_over
       - game_mode (PICK_FACTION_BLUE=0..PLAYING=4)
       - factions, sel_faction_idx, faction_id, leader_placed, etc.
       - recruit_mode, recruit_palette, etc.
       - Camera state: board_origin, camera_offset, camera_min/max, drag state, lerp state
       - inspect_unit_id

    3. **Hex math** (replaces Godot TileMap):
       - `hex_to_pixel(col, row)` — returns center x,y for odd-r pointy-top hex:
         `x = HEX_CELL_W * (col + 0.5 * (row % 2))`
         `y = HEX_CELL_H * 0.75 * row`
       - `pixel_to_hex(px, py)` — inverse: convert mouse position to hex col,row
         Account for camera offset (subtract board_origin + camera_offset before converting).
         Use the standard approach: estimate from rectangular grid, then check distance to candidate hex centers.
       - `hex_polygon(cx, cy, radius)` — returns flat table of x,y pairs for pointy-top hexagon (6 vertices, angle starting at -π/6)

    4. **love.load()** — Mirrors _ready():
       - Create engine via norrust.new()
       - Construct data_path (project root / data)
       - Call norrust.load_data, norrust.load_board (contested.toml, seed=42), norrust.load_factions
       - Load units: norrust.load_units (contested_units.toml)
       - Parse faction list from norrust.get_faction_ids_json
       - Create fonts at required sizes (9, 11, 12, 14, 15, 18, 32)
       - Compute board_origin and camera clamp bounds (replaces _center_camera)

    5. **love.update(dt)** — Mirrors _process(delta):
       - Arrow key continuous panning (CAMERA_PAN_SPEED * dt)
       - Camera lerp toward target (lerp factor CAMERA_LERP_SPEED * dt)
       - Clamp camera offset to bounds

    6. **love.draw()** — Mirrors _draw():
       - Parse state (norrust.get_state)
       - Build tile_colors map from state.terrain
       - Apply camera transform: love.graphics.push() + translate(board_origin + camera_offset)
       - Draw terrain hexes (filled polygons with terrain color)
       - Draw reachable hex highlights (yellow, alpha 0.35)
       - Draw selected unit outline (white polyline, width 2.5)
       - Draw units (circles with faction color, name/HP/XP labels, advancement arc)
       - love.graphics.pop() — back to screen space
       - Draw HUD (top-left: turn/faction/gold/ToD)
       - Draw unit panel or recruit panel (right sidebar, 200px wide)
       - Draw setup HUD (faction picker / setup panels)
       - Draw game over overlay if applicable

    7. **love.keypressed(key)** — Mirrors keyboard handling in _input():
       - "e": end turn + AI trigger
       - "a": advance selected unit
       - "r": toggle recruit mode
       - "1"-"9": faction/recruit selection
       - "return": confirm setup phase
       - Arrow keys handled in love.update (continuous, not one-shot)

    8. **love.mousepressed(x, y, button)** — Mirrors mouse press in _input():
       - Convert screen coords to hex via pixel_to_hex (accounting for camera)
       - Same logic: sidebar check → off-board drag → recruit placement → attack → move → select → inspect → drag start

    9. **love.mousereleased(x, y, button)** — End drag

    10. **love.mousemoved(x, y, dx, dy)** — Drag-to-pan while button held

    **Key Love2D API mappings from Godot:**
    - `draw_polygon()` → `love.graphics.polygon("fill", vertices)`
    - `draw_polyline()` → `love.graphics.polygon("line", vertices)` with `love.graphics.setLineWidth()`
    - `draw_string()` → `love.graphics.print(text, x, y)` or `love.graphics.printf()` for alignment
    - `draw_arc()` → Use `love.graphics.arc("line", x, y, radius, angle1, angle2, segments)`
    - `draw_circle()` → `love.graphics.circle("fill", x, y, radius)`
    - `draw_rect()` → `love.graphics.rectangle("fill", x, y, w, h)`
    - `Color(r, g, b, a)` → `love.graphics.setColor(r, g, b, a)`
    - `get_viewport_rect().size` → `love.graphics.getDimensions()`

    **Important differences from Godot:**
    - Love2D indices are 1-based (Lua convention)
    - Love2D font sizes set via `love.graphics.newFont(size)`; create fonts in love.load
    - love.graphics.setColor must be called BEFORE each draw call (no per-call color param)
    - love.keyboard.isDown("left") for continuous key state in love.update
    - JSON state parsed via norrust module's JSON decoder
    - No TileMap coordinate conversion — use hex_to_pixel/pixel_to_hex functions
    - Polygon vertices as flat array: {x1,y1, x2,y2, ...} (not packed arrays)

    **Do NOT:**
    - Add any features not in game.gd
    - Change any game constants or logic
    - Modify any Rust code
    - Use external Lua libraries beyond Love2D stdlib
