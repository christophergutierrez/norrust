- `norrust_love/main.lua` — slimmed to ~950 lines, love.* input callbacks delegate to input module

<task type="auto">
  <name>Task 1: Create input.lua module and wire dispatchers</name>
  <files>norrust_love/input.lua, norrust_love/main.lua</files>
  <action>
    Create `norrust_love/input.lua` that exports handler functions. The module receives a context table (`ctx`) containing all state and helper references it needs.

    **Pattern:**
    ```lua
    local M = {}

    function M.init(ctx)
        M.ctx = ctx
    end

    function M.keypressed(key) ... end
    function M.mousepressed(sx, sy, button) ... end
    function M.mousereleased(x, y, button) ... end
    function M.mousemoved(sx, sy, dx, dy) ... end
    function M.wheelmoved(x, y) ... end

    return M
    ```

    **The ctx table** (built once in love.load and passed to input.init) contains:
    - Context tables: `scn`, `sel`, `ghost`, `campaign`, `dlg`, `camera` (direct references — mutations in input.lua are visible in main.lua)
    - State refs that can't be mutated via table: wrap in a `state` table: `state = {game_mode, game_over, winner_faction, next_unit_id, status_message, status_timer, faction_id, leader_placed, sel_faction_idx}`
    - Since Lua tables are passed by reference, mutating `ctx.sel.unit_id = 5` in input.lua changes the value in main.lua. But scalars like `game_mode` need a wrapper table.
    - Shared tables: `shared`, `combat_state`, `pending_anims`, `factions`, `FACTION_COLORS`
    - Modules: `norrust`, `hex`, `events`, `save`, `roster_mod`
    - Constants: `SCENARIOS`, `CAMPAIGNS`, `PLAYING`, `PICK_SCENARIO`, `PICK_FACTION_BLUE`, `PICK_FACTION_RED`, `SETUP_BLUE`, `SETUP_RED`
    - Helper functions from main.lua: `engine` (getter), `get_viewport`, `screen_to_game`, `int`, `clamp`, `center_camera`, `clear_selection`, `cancel_ghost`, `cancel_combat_preview`, `is_ranged_attack`, `commit_ghost_move`, `execute_attack`, `check_game_over`, `build_unit_pos_map`, `unit_max_range`, `select_unit`, `get_attackable_enemies`, `ghost_attackable_set`, `call_load_scenario`, `call_load_campaign_scenario`, `fire_hex_entered`, `faction_index_for_mode`

    **Critical: engine access.** `engine` is a plain local in main.lua that gets set in love.load. Since it's not a table field, input.lua can't see mutations. Solution: put engine in a holder table `ctx.ref = {engine = engine}` and update it when engine changes, OR make ctx.engine a function: `ctx.engine = function() return engine end`. The function approach is cleanest.

    Actually, simpler: put ALL scalar state vars that input handlers read/write into a single `state` table in main.lua:
    ```lua
    local state = {
        game_mode = PICK_SCENARIO,
        game_over = false,
        winner_faction = -1,
        next_unit_id = 1,
        status_message = nil,
        status_timer = 0,
        sel_faction_idx = 0,
    }
    ```
    Then `engine`, `faction_id`, `leader_placed`, `factions` stay as separate locals in main.lua but get passed to ctx. For `engine`, which changes after love.load, use a holder: `ctx.engine_ref = {engine}` or a function.

    **Simplest approach:** Don't create a new state table — just pass engine as `ctx.engine` and update it after love.load sets it. Since ctx is built in love.load AFTER engine is created, `ctx.engine = engine` captures the correct value. Engine never changes after that (load_board creates new GameState but engine pointer stays same thanks to Rust Box).

    **Implementation steps:**

    1. Move these functions from main.lua to input.lua:
       - love.keypressed body → M.keypressed (including build_save_campaign_ctx local, shared.handle_sidebar_button)
       - love.mousepressed body → M.mousepressed
       - love.mousereleased body → M.mousereleased
       - love.mousemoved body → M.mousemoved
       - love.wheelmoved body → M.wheelmoved

    2. In input.lua, all references to main.lua locals become ctx.X:
       - `engine` → `ctx.engine`
       - `game_mode` → `ctx.game_mode` (needs to be in a mutable table)
       - `game_over` → `ctx.game_over`
       - `winner_faction` → `ctx.winner_faction`
       - etc.

    3. **Mutable scalar problem:** When input.lua does `ctx.game_mode = PLAYING`, this works if ctx is a table — the field update is visible to main.lua because both reference the same table. So the approach is:
       - Build ctx in love.load with ALL mutable scalars as fields
       - main.lua reads them back as `ctx.game_mode` instead of `game_mode`
       - This means main.lua also needs to change its reads to use ctx.field for any scalar that input.lua mutates

    4. **Alternative (simpler for main.lua):** Create a small `vars` table for just the mutable scalars that input handlers write:
       ```lua
       local vars = {
           game_mode = PICK_SCENARIO,
           game_over = false,
           winner_faction = -1,
           next_unit_id = 1,
           status_message = nil,
           status_timer = 0,
           sel_faction_idx = 0,
       }
       ```
       Pass `vars` to input.lua. main.lua reads `vars.game_mode` everywhere. This is the same pattern as Phase 78's context tables — just one more table for "global" mutable state.

    5. In main.lua, replace love.keypressed/mousepressed/etc with thin dispatchers:
       ```lua
       function love.keypressed(key)
           input.keypressed(key)
       end
       ```

    **Important:**
    - `faction_id` and `leader_placed` are already tables (arrays), so mutations via ctx work naturally
    - `factions` is already a table
    - Context tables (scn, sel, ghost, campaign, dlg, camera) are already tables — mutations propagate
    - Only pure scalars (game_mode, game_over, winner_faction, next_unit_id, status_message, status_timer, sel_faction_idx) need the vars wrapper

    **Do NOT:**
    - Change draw.lua or any other module
    - Rename any variables — just move code and adjust references
    - Change game logic or behavior
    - Extract helper functions (cancel_ghost, commit_ghost_move, etc.) — those stay in main.lua and are passed via ctx
