<task type="auto">
  <name>Task 1: Melee approach animation in execute_attack</name>
  <files>norrust_love/main.lua</files>
  <action>
    Modify `execute_attack(attacker_id, defender_id, is_ranged)` to add a melee approach animation:

    1. If `is_ranged` is true, keep current behavior (trigger anims + apply attack immediately).

    2. If `is_ranged` is false (melee):
       a. Look up attacker and defender positions from current game state.
       b. Compute a midpoint ~40% of the way from attacker hex center to defender hex center.
       c. Build a 2-waypoint "path" for pending_anims.move: [{attacker hex}, {midpoint as fake waypoint}].
          - Since draw.lua interpolates pixel positions between path waypoints, use pixel math:
            create a synthetic path entry with fractional col/row that maps to the desired pixel midpoint.
          - Actually simpler: don't use pending_anims.move for this. Instead, add a new field
            `pending_anims.combat_slide` = {uid, start_x, start_y, target_x, target_y, t, speed, phase, on_complete}
            where phase = "approach" or "return".
       d. In the approach phase: slide attacker from start toward target (~40% distance).
       e. When approach completes: trigger attack anims, apply attack, check death.
       f. After a brief pause (0.3s), start return phase: slide attacker back to start position.
       g. When return completes: call on_complete if any, clear combat_slide.

    3. Add `pending_anims.combat_slide` tick in `love.update(dt)` (after the existing move_anim tick):
       - Advance t by speed * dt
       - If approach phase and t >= 1.0: trigger anims, apply attack, start pause timer
       - After pause: switch to return phase (reset t, swap start↔target)
       - If return phase and t >= 1.0: clear combat_slide, call on_complete

    4. In draw.lua, add combat_slide interpolation (same pattern as move_anim):
       - If `ctx.combat_slide` and `ctx.combat_slide.uid == uid`: override cx, cy with lerped position
       - Lerp between start and target using t value

    5. Pass `pending_anims.combat_slide` through ctx as `combat_slide` (same as move_anim pattern).

    6. Add input blocking: include `pending_anims.combat_slide` in the existing input guard checks.

    7. For ghost-move+attack: the move callback already calls execute_attack. The approach animation
       will chain naturally — move finishes, callback fires, execute_attack starts approach animation.
       Input stays blocked because combat_slide is checked.

    **Tuning:**
    - Approach speed: ~6 (takes ~0.17s to reach 40% point)
    - Pause at contact: 0.3s (let attack anims play)
    - Return speed: ~6 (same as approach)
    - Total melee attack time: ~0.6-0.7s
