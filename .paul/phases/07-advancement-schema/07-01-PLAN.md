<objective>
## Goal
Extend data definitions and runtime structs to carry unit advancement information.
TOML files gain `level`, `experience`, `advances_to`; `Unit` gains `xp`, `xp_needed`,
`advancement_pending`; `UnitSnapshot` JSON exposes all three for GDScript and AI clients.
New level-2 TOML files (hero, ranger) are created so the advancement chain is complete
end-to-end.

## Purpose
Unit advancement is the defining feature of v0.3. All subsequent logic (Phase 8: XP gain,
`Action::Advance`) and presentation (Phase 9: XP HUD, advance indicator) depend on these
fields existing in the schema, runtime struct, and JSON snapshot. This plan lays the
schema layer so Phase 8 can implement mechanics without touching data definitions.

## Output
- `schema.rs`: `UnitDef` has `level: u8`, `experience: u32`, `advances_to: Vec<String>`
  (all `#[serde(default)]` so existing TOML without these fields still loads)
- `data/units/fighter.toml` + `archer.toml`: `level`, `experience`, `advances_to` added
- `data/units/hero.toml` + `ranger.toml`: new level-2 unit definitions
- `unit.rs`: `Unit` has `xp: u32`, `xp_needed: u32`, `advancement_pending: bool`
- `gdext_node.rs`: `place_unit_at()` initialises `xp_needed` from `def.experience`
- `snapshot.rs`: `UnitSnapshot` exposes `xp`, `xp_needed`, `advancement_pending`;
  `from_game_state()` maps the new fields; snapshot tests updated
</objective>
