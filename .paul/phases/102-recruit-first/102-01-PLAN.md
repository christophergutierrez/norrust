<objective>
## Goal
Make the AI turn planner recruit-aware: simulate all affordable recruitment before planning unit movement, so the leader naturally stays on keep while gold remains and newly recruited units participate in the turn's movement planning.

## Purpose
Currently recruitment happens in FFI (`norrust_ai_recruit`) before `plan_full_turn`, but the turn planner doesn't see recruits when evaluating orderings. The leader's `should_leader_stay` correctly keeps it on keep, but there's no integration: the planner doesn't know recruits exist, and doesn't plan their movement. More importantly, the planner can't evaluate "stay and recruit" vs "leave keep" because recruitment is external.

Moving recruitment simulation into `plan_full_turn` means: (1) recruited units participate in movement planning, (2) the leader naturally stays on keep because the planner sees the value of recruiting, (3) all gold is spent before anyone moves.

## Output
Modified `plan_full_turn` and `run_turn_ordering` in `norrust_core/src/ai.rs` with integrated recruitment simulation.
</objective>

## Key Facts
- `norrust_ai_recruit` in ffi.rs handles actual recruitment: finds leader on keep, finds empty castle slots, picks affordable units round-robin, calls apply_recruit
- `plan_full_turn` clones state and tries K=5 orderings via `run_turn_ordering`
- `run_turn_ordering` takes a unit order list and simulates moves/attacks on a clone
- `should_leader_stay` returns true when leader is on keep AND has gold AND has empty castle slots
- The Lua side calls `ai_recruit(engine, fid)` then `ai_plan_turn(engine, faction)` — recruitment already happens first in practice
- `apply_recruit(state, unit, destination, cost)` in game_state.rs handles the actual placement + gold deduction
- Unit creation needs UnitDef from registry — but plan_full_turn doesn't have registry access
- State has: `state.gold[faction]`, `state.next_unit_id`
- Castle hexes have `terrain_id == "castle"`, keep hexes have `terrain_id == "keep"`

<task type="auto">
  <name>Task 1: Add recruit simulation to run_turn_ordering</name>
  <files>norrust_core/src/ai.rs</files>
  <action>
    Add a `recruit_defs` parameter to `plan_full_turn` and `run_turn_ordering` — a slice of `(cost: u32, movement: u32)` tuples representing available recruit types (cost and movement stat for the simulated placeholder unit).

    In `run_turn_ordering`, before the unit movement loop:
    1. Find the leader in the clone. If leader is on a keep hex and faction has gold:
    2. Loop: while gold >= cheapest recruit cost AND there are empty castle hexes adjacent to the keep:
       a. Pick the recruit type using round-robin (cycle through recruit_defs by index)
       b. Find an empty castle hex adjacent to the keep
       c. Create a minimal placeholder Unit: `Unit::new(clone.next_unit_id, "recruit", movement, faction)` with `max_hp = movement` (we only need it to move; it won't fight meaningfully in 1-ply sim)
       d. Give it a basic melee attack (damage=5, strikes=2, range="melee") so it can participate in combat scoring
       e. Set `unit.movement = recruit_def.movement` and `unit.movement_costs` to empty HashMap (defaults to cost 1 everywhere)
       f. Place it: `clone.place_unit(unit, castle_hex)`
       g. Deduct gold: `clone.gold[faction as usize] -= cost`
       h. Track the recruited unit IDs
    3. Append recruited unit IDs to the unit ordering (after existing units)
    4. The existing movement loop will then plan actions for these recruited units too

    Update `plan_full_turn` to accept and pass through `recruit_defs`.
    Update `ai_take_turn` and `ai_plan_turn` to pass `recruit_defs` (can be empty slice initially — actual recruit info comes from FFI).

    **Important:** The simulated recruits are for PLANNING only — the real recruitment still happens via `norrust_ai_recruit` in FFI. The planner just needs to see that recruits will exist so it can evaluate the board state correctly and keep the leader on keep.

    For now, pass an empty `&[]` for recruit_defs from ai_take_turn/ai_plan_turn. Task 2 will wire up the actual recruit info.
  </action>
