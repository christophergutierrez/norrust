<objective>
## Goal
Write the asset format specification document and build the asset loader module for Love2D with fallback rendering — sprites when available, colored circles/polygons when not.

## Purpose
Establishes the foundation for the entire v1.4 Visual Asset System. The spec document defines conventions that all future phases follow. The loader module provides the runtime infrastructure so terrain tiles (Phase 32) and unit sprites (Phase 33) can be loaded and rendered immediately.

## Output
- `docs/ASSET-SPEC.md` — comprehensive asset format specification
- `norrust_love/assets.lua` — asset loader module (terrain tiles + unit sprites + fallback)
- `norrust_love/main.lua` — updated to use assets.lua for terrain hex rendering and unit drawing
</objective>

## Key v1.4 Decisions
- High-res source art (256x256+), scaled at runtime
- TOML sidecar per unit (sprite.toml) in assets/units/{def_id}/
- Terrain images flat by terrain_id in assets/terrain/
- Fallback rendering: sprite when available, colored polygon/circle when not
- No Rust changes — purely client-side

<task type="auto">
  <name>Task 1: Write asset specification document</name>
  <files>docs/ASSET-SPEC.md</files>
  <action>
    Create comprehensive asset specification covering:

    1. **Directory Structure:**
       - `assets/terrain/{terrain_id}.png` — hex terrain tiles
       - `assets/units/{def_id}/` — per-unit directories
       - `assets/units/{def_id}/sprite.toml` — metadata sidecar
       - `assets/units/{def_id}/idle.png` — static/spritesheet
       - `assets/units/{def_id}/attack-{type}.png` — attack animation
       - `assets/units/{def_id}/death.png` — death animation
       - `assets/units/{def_id}/portrait.png` — sidebar portrait

    2. **Terrain Tile Format:**
       - PNG, hex-shaped (transparent outside hex boundary) or rectangular (clipped by hex mask)
       - Recommended source size: 256x256 or larger
       - Scaled at runtime to match HEX_RADIUS (currently 96px)
       - Named by terrain_id from data/terrain/*.toml (flat.png, forest.png, hills.png, etc.)

    3. **sprite.toml Schema:**
       ```toml
       id = "spearman"

       [idle]
       file = "idle.png"
       frame_width = 256
       frame_height = 256
       frames = 4
       fps = 4
       anchor_x = 128
       anchor_y = 200

       [attacks.melee]
       file = "attack-melee.png"
       frame_width = 256
       frame_height = 256
       frames = 6
       fps = 8

       [death]
       file = "death.png"
       frame_width = 256
       frame_height = 256
       frames = 4
       fps = 6

       [portrait]
       file = "portrait.png"
       ```

    4. **Team Coloring:**
       - Colored underlay circle behind sprite (existing faction circle becomes background)
       - Sprite drawn on top, centered at anchor point
       - Future: tint shader for recoloring marked regions

    5. **Animation States:**
       - idle (default loop), attack-melee, attack-ranged, defend, death
       - Spritesheets: horizontal strip, equal-width frames
       - FPS defined per animation in sprite.toml

    6. **Naming Conventions:**
       - terrain_id matches TOML id field (flat, forest, hills, mountains, etc.)
       - def_id matches TOML id field in data/units/ (fighter, spearman, etc.)
       - All lowercase, hyphens for multi-word (attack-melee, not attackMelee)

    7. **Pipeline Workflow:**
       - Generate with Nano Banana → manual cleanup → place in assets/ → validate with viewer
  </action>
  <verify>File exists at docs/ASSET-SPEC.md with all 7 sections</verify>
  <done>AC-1 satisfied: specification document covers all asset conventions</done>

<task type="auto">
  <name>Task 2: Create assets.lua loader module</name>
  <files>norrust_love/assets.lua</files>
  <action>
    Create `norrust_love/assets.lua` module that:

    1. **load_terrain_tiles(base_path)**
       - Scans `{base_path}/terrain/` for PNG files
       - Loads each as love.graphics.newImage
       - Returns table keyed by terrain_id (filename without .png)
       - Gracefully handles missing directory (returns empty table)

    2. **load_unit_sprites(base_path)**
       - Scans `{base_path}/units/` for directories
       - For each directory with idle.png, loads the image
       - Returns table keyed by def_id (directory name)
       - Each entry: { idle = Image, portrait = Image|nil }
       - Gracefully handles missing directory (returns empty table)
       - NOTE: Full spritesheet/animation loading deferred to Phase 33 (anim8)
       - For now, idle.png treated as single static image

    3. **draw_terrain_hex(terrain_tiles, terrain_id, cx, cy, hex_radius, fallback_color)**
       - If terrain_tiles[terrain_id] exists: draw scaled image centered at cx,cy clipped to hex
       - Else: draw colored polygon using fallback_color (current behavior)
       - Scaling: image scaled to fit hex diameter (hex_radius * 2)

    4. **draw_unit_sprite(unit_sprites, def_id, cx, cy, hex_radius, faction, alpha)**
       - If unit_sprites[def_id] exists:
         - Draw faction-colored circle as underlay (same size as current)
         - Draw sprite centered on hex, scaled to fit within hex
       - Else: return false (caller draws current circle+text fallback)

    Module returns table: { load_terrain_tiles, load_unit_sprites, draw_terrain_hex, draw_unit_sprite }

    Implementation notes:
    - Use love.filesystem.getDirectoryItems() for scanning
    - Use love.graphics.newImage() for loading
    - Use love.graphics.stencil or scissor for hex clipping of terrain (or accept rectangular for now and refine in Phase 32)
    - Scale with sx, sy params on love.graphics.draw()
    - All paths relative to Love2D's source directory
  </action>
  <verify>Module loads without errors: `require("assets")` succeeds in Love2D</verify>
  <done>AC-2, AC-3 partially satisfied: loader infrastructure exists with fallback paths</done>
</task>
