<objective>
## Goal
Replace the Lua-side `commit_deployment` logic with a single Rust FFI call. The engine handles keep/castle finding, slot validation, and veteran placement — Lua only sends which veterans the player selected.

## Purpose
`commit_deployment` in campaign_client.lua is the last major Lua gameplay logic: it duplicates `find_keep_and_castles` (already in campaign.rs), iterates slots, and calls `place_veteran_unit` per veteran. This phase migrates it to a single FFI call, completing the v5.0 milestone — Rust engine owns all gameplay, Lua is purely presentation/input.

## Output
- New FFI function `norrust_campaign_commit_deployment` that takes deployed veteran indices and places them
- `commit_deployment` in campaign_client.lua replaced with single FFI call
- `find_keep_and_castles` removed from campaign_client.lua (Rust already has it)
- Dead ctx fields (`campaign_veterans`, `campaign_roster`, `roster_mod`) removed from input_deploy.lua
</objective>

<task type="auto">
  <name>Task 1: Create norrust_campaign_commit_deployment FFI</name>
  <files>norrust_core/src/ffi.rs</files>
  <action>
    Create `norrust_campaign_commit_deployment(engine, deployed_json) -> *const c_char`:
    - Takes a JSON string: array of indices into the engine's `campaign.veterans` that the player deployed (e.g., `[0, 2, 3]`)
    - Reads CampaignState.veterans to get the VeteranUnit entries at those indices
    - Calls `find_keep_and_castles(state, 0)` to get placement hexes
    - Builds slot list: keep first, then adjacent castles
    - For each deployed veteran, finds next unoccupied slot, creates unit via `unit_from_registry`, sets hp/xp/xp_needed/advancement_pending, places it, maps engine ID to roster UUID via `cs.map_id`
    - Returns JSON: `{"status":"ok","placed":N}` or `{"status":"error","message":"..."}`

    This reuses the same placement logic as the auto-placement path in `norrust_campaign_load_next_scenario` (lines 1782-1852), just driven by user-selected indices instead of all veterans.

    Also clear `cs.id_map` before placement (same as auto-placement does).

    Avoid: duplicating unit creation logic — reuse `unit_from_registry` exactly as `load_next_scenario` does.
  </action>
  <verify>cargo test --lib -q (in norrust_core/) — compiles and all existing tests pass</verify>
  <done>AC-1 satisfied: single FFI function for veteran deployment placement</done>
