<objective>
## Goal
Replace the greedy `score_attack` unit decision with 1-ply lookahead: for each unit, clone the state, try all reachable (move, attack) combinations, evaluate each resulting state with `evaluate_state`, and pick the action producing the best score.

## Purpose
The current AI scores individual attacks in isolation (expected damage ratio). This misses terrain positioning, kill securing across units, and defensive value of moves. 1-ply lookahead evaluates the *resulting board state* after each candidate action, producing smarter decisions.

## Output
Modified `ai_plan_turn` in `norrust_core/src/ai.rs` using evaluate_state for per-unit action selection. `ai_take_turn` updated to match.
</objective>

## Key Facts
- `ai_plan_turn` clones state, iterates units, picks best (dest, target) using score_attack, applies actions to clone, returns Vec<ActionRecord>
- `ai_take_turn` does the same but mutates real state directly
- `evaluate_state(state, faction)` returns f32 — higher is better for faction
- GameState derives Clone — cloning for simulation is cheap enough
- Both functions share identical logic (duplicated) — leader discipline, candidate enumeration, enemy scanning, scoring, march fallback
- Current scoring: `score_attack(unit, dest, enemy, epos, state)` — per-attack expected damage ratio
- New scoring: clone state → apply Move → apply Attack → `evaluate_state(clone, faction)` → compare
- For "move only" (no attack possible): clone → apply Move → evaluate — enables positioning moves
- March fallback (move toward nearest enemy) can also use evaluate: pick the reachable hex that produces the best evaluated state
- Combat is deterministic in apply_action (uses RNG) — evaluate after single apply gives the actual outcome, not expected value. This is fine for 1-ply; the RNG outcome is what would actually happen.

<task type="auto">
  <name>Task 1: Extract shared unit action planner with lookahead</name>
  <files>norrust_core/src/ai.rs</files>
  <action>
    Create a shared helper that both `ai_take_turn` and `ai_plan_turn` call for per-unit decisions:

    1. Add `fn plan_unit_action(state: &GameState, uid: u32, faction: u8) -> Option<(Hex, Option<u32>)>` that returns (destination, optional_attack_target). This replaces the duplicated scoring loop in both functions.

    2. Inside `plan_unit_action`:
       a. Get unit's start hex, movement, compute reachable candidates (same as current code)
       b. For each candidate hex, enumerate attackable enemies (same adjacency/range logic as current)
       c. For each (candidate, enemy) pair:
          - Clone state
          - Apply Move to clone (if candidate != start)
          - Apply Attack to clone
          - Score with `evaluate_state(&clone, faction)`
          - Track best score + best (dest, target)
       d. Also evaluate "move only" for each candidate (no attack):
          - Clone state
          - Apply Move to clone
          - Score with `evaluate_state(&clone, faction)`
          - If score > best_score, update best
       e. Also evaluate "stay put" (no move, no attack) as baseline:
          - Score = `evaluate_state(state, faction)` (no clone needed)
          - Only pick a move/attack if it improves over staying
       f. Return best (dest, target) or None if staying is best

    3. Update `ai_plan_turn` to call `plan_unit_action` instead of the inline scoring loop. Keep leader discipline and march logic. Replace the score_attack block with:
       ```
       if let Some((dest, target)) = plan_unit_action(&clone, uid, faction) {
           // apply move + attack to clone, record actions
       }
       ```

    4. Update `ai_take_turn` similarly — call `plan_unit_action(state, uid, faction)` then apply the chosen action to real state.

    5. Remove the march fallback — the move-only evaluation in plan_unit_action now handles positioning (it will naturally pick the hex with the best evaluated state, which should favor proximity to enemies via the evaluation function's positional scoring).

    6. Keep `score_attack` — it may still be useful for tie-breaking or future phases. Just remove it from the decision loop.

    Note: The clone-per-candidate approach means N_candidates × N_enemies clones per unit. For typical boards (8×5 to 16×10) with movement=5, this is ~20-50 candidates × ~5-10 enemies = 100-500 clones per unit, 5-10 units = 500-5000 clones per turn. GameState clone is cheap (HashMap clone). This should be well under 100ms per turn.
