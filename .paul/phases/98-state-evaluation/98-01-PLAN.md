<objective>
## Goal
Add a pure `evaluate_state(state, faction) -> f32` function that scores a GameState from a given faction's perspective, considering HP totals, unit count, village control, positional value, gold advantage, and leader safety.

## Purpose
Phase 99 (1-Ply Lookahead) needs an evaluation function to compare candidate game states after simulated moves. This is the foundation for all future AI improvements — without it, the AI can only score individual attacks (score_attack), not holistic board positions.

## Output
New `evaluate_state` function in `norrust_core/src/ai.rs` with unit tests validating each scoring component.
</objective>

<context>
## Source Files
@norrust_core/src/ai.rs (current AI logic — score_attack, leader helpers, ai_take_turn, ai_plan_turn)
@norrust_core/src/game_state.rs (GameState struct — units, positions, gold, village_owners, objective_hex, check_winner)

## Key Facts
- GameState has: units (HashMap<u32, Unit>), positions (HashMap<u32, Hex>), board, gold[2], village_owners (HashMap<Hex, i8>), objective_hex, max_turns
- Unit has: hp, max_hp, faction, abilities, attacks, level, movement, defense, resistances, alignment
- Village ownership: village_owners maps hex → faction (i8::MAX = unowned)
- Objective hex: if set, faction 0 wins by reaching it
- check_winner() returns Some(faction) if game is decided
- Board has: width, height, tile_at(hex) → Option<&Tile>
- This function will be consumed by Phase 99's lookahead (clone state → apply action → evaluate)
</context>

<acceptance_criteria>

## AC-1: Evaluate function returns higher score for advantageous states
```gherkin
Given a GameState where faction A has more total HP than faction B
When evaluate_state is called for faction A
Then the returned score is higher than when faction B has the HP advantage
```

## AC-2: Village control contributes to score
```gherkin
Given two otherwise identical GameStates differing only in village ownership
When evaluate_state is called
Then the state with more villages owned by the evaluated faction scores higher
```

## AC-3: Won/lost states produce extreme scores
```gherkin
Given a GameState where check_winner() returns a winner
When evaluate_state is called for the winning faction
Then the score is a large positive value (f32::MAX or similar sentinel)
And the score for the losing faction is a large negative value
```

## AC-4: Positional value rewards proximity to objectives/enemies
```gherkin
Given a GameState with an objective hex
When evaluate_state is called for the attacking faction
Then units closer to the objective contribute more positional value
```

## AC-5: Leader safety contributes to score
```gherkin
Given two states where the leader is at different HP levels
When evaluate_state is called
Then the state with a healthier leader scores higher
```

  <name>Task 1: Implement evaluate_state function</name>
  <files>norrust_core/src/ai.rs</files>
  <action>
    Add a new section `// ── State evaluation ──` in ai.rs (after the leader discipline helpers, before the core AI turn logic section).

    Implement `pub fn evaluate_state(state: &GameState, faction: u8) -> f32` with these scoring components:

    1. **Terminal state check**: If `check_winner()` returns Some(winner):
       - Return `f32::MAX` if winner == faction
       - Return `f32::MIN` if winner != faction

    2. **HP advantage** (weight ~2.0): Sum of own units' HP minus sum of enemy units' HP. Scaled by 2.0 to make HP the primary factor.

    3. **Unit count advantage** (weight ~10.0): (own_unit_count - enemy_unit_count) × 10.0. Losing a unit is significant beyond just HP loss.

    4. **HP ratio bonus** (weight ~5.0): own_total_hp / (own_total_hp + enemy_total_hp) × 100.0 × 5.0. Rewards having a larger share of total HP on the board. Guards against division by zero.

    5. **Village control** (weight ~8.0): Count of villages owned by faction × 8.0. Villages provide income and healing — controlling them is strategically valuable.

    6. **Gold advantage** (weight ~0.5): (own_gold - enemy_gold) × 0.5. Gold matters but less than board position since it's potential, not actualized.

    7. **Positional value — objective proximity** (weight ~3.0): If objective_hex is set, for each own unit: (board_diameter - distance_to_objective) × 3.0 / board_diameter. Only for faction 0 (attacker). For faction 1, reward distance of own units FROM objective (defensive positioning — being near it to block).

    8. **Leader safety bonus** (weight ~15.0): If own leader exists, add leader.hp as f32 / leader.max_hp as f32 × 15.0. Losing the leader is catastrophic. If enemy leader is low HP, add a bonus for opportunity to kill it.

    Keep the function simple and readable. Each component should be clearly commented with its purpose. The weights don't need to be perfect — Phase 99 will expose whether the eval function produces sensible move choices, and weights can be tuned later.
  </action>
