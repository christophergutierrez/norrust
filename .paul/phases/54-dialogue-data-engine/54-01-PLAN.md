<objective>
## Goal
Create a dialogue TOML schema, Rust loader, runtime state tracker, and FFI query functions so that Love2D can ask "what dialogue should fire now?" given the current turn, phase, and event.

## Purpose
Enables scenario authors to attach narrator dialogue to game events without modifying engine mechanics. First half of the dialogue system — data and engine only, no rendering.

## Output
- `dialogue.rs` — schema, loader, runtime DialogueState with fired tracking
- Schema additions in `schema.rs` for the DialogueDef TOML structure
- FFI functions: `norrust_load_dialogue`, `norrust_get_dialogue`
- Sample dialogue file: `scenarios/crossing_dialogue.toml`
- Integration tests proving load + query + one-shot semantics
</objective>

## AC-2: Turn-Based Query Returns Matching Dialogue
```gherkin
Given a loaded dialogue set with entries for "turn_start" at turn 1 and "turn_end" at turn 2
When get_pending_dialogue() is called with trigger="turn_start", turn=1
Then it returns the matching entry text

```

## AC-3: One-Shot Semantics
```gherkin
Given a dialogue entry that has already been returned by get_pending_dialogue()
When get_pending_dialogue() is called again with the same trigger and turn
Then the entry is NOT returned again (fired entries are tracked)
```

```gherkin
And norrust_get_dialogue() is called with trigger="turn_start" and turn=1
Then a JSON string is returned containing the matching dialogue text
```

<task type="auto">
  <name>Task 1: Dialogue schema, loader, and runtime state</name>
  <files>norrust_core/src/dialogue.rs, norrust_core/src/lib.rs, scenarios/crossing_dialogue.toml</files>
  <action>
    Create norrust_core/src/dialogue.rs with:

    1. **DialogueEntry struct** (Deserialize):
       - `id: String` — unique identifier within the file
       - `trigger: String` — "scenario_start", "turn_start", "turn_end" (gameplay triggers added in Phase 57)
       - `turn: Option<u32>` — which turn this fires on (None = any turn, for scenario_start)
       - `faction: Option<u8>` — which faction's turn (None = either faction)
       - `text: String` — narrator dialogue text

    2. **DialogueFile struct** (Deserialize wrapper):
       - `dialogue: Vec<DialogueEntry>` — the `[[dialogue]]` array from TOML

    3. **DialogueState struct** (runtime):
       - `entries: Vec<DialogueEntry>` — loaded from file
       - `fired: HashSet<String>` — IDs of entries already returned
       - Methods:
         - `load(path: &Path) -> Result<Self, String>` — read TOML, parse, return fresh state
         - `get_pending(&mut self, trigger: &str, turn: u32, faction: u8) -> Vec<&DialogueEntry>` — return matching unfired entries, mark them fired
         - `reset(&mut self)` — clear fired set (for scenario restart)

    4. Matching logic in get_pending:
       - Entry matches if: entry.trigger == trigger
       - AND (entry.turn is None OR entry.turn == Some(turn))
       - AND (entry.faction is None OR entry.faction == Some(faction))
       - AND entry.id is NOT in self.fired
       - After collecting matches, insert all matched IDs into self.fired

    5. Add `pub mod dialogue;` to lib.rs

    6. Create scenarios/crossing_dialogue.toml with sample entries:
       - scenario_start: "Your forces have reached the river crossing. The enemy keep lies to the east — you must reach it before reinforcements arrive."
       - turn_start turn=3: "Your scouts report enemy patrols moving south. Press forward before they consolidate."
       - turn_end turn=5: "Dusk approaches. Your troops grow weary, but the enemy keep is within sight."

    7. Unit tests in dialogue.rs:
       - test_load_dialogue: load crossing_dialogue.toml, verify entry count and fields
       - test_get_pending_filters: verify trigger+turn matching
       - test_one_shot: verify second call returns empty
       - test_reset: verify reset clears fired state
  </action>
  <verify>cargo test -p norrust_core -- dialogue -- all 4 tests pass</verify>
  <done>AC-1, AC-2, AC-3, AC-5 satisfied</done>
</task>

<task type="auto">
  <name>Task 2: FFI bridge for dialogue loading and querying</name>
  <files>norrust_core/src/ffi.rs, norrust_core/tests/dialogue.rs</files>
  <action>
    1. Add `dialogue_state: Option<DialogueState>` field to NorRustEngine struct.
       Initialize to None in NorRustEngine::new().

    2. Add FFI function `norrust_load_dialogue`:
       - Signature: `extern "C" fn norrust_load_dialogue(engine: *mut NorRustEngine, path: *const c_char) -> i32`
       - Load dialogue from path via DialogueState::load()
       - Store in engine.dialogue_state
       - Return 1 on success, 0 on failure

    3. Add FFI function `norrust_get_dialogue`:
       - Signature: `extern "C" fn norrust_get_dialogue(engine: *mut NorRustEngine, trigger: *const c_char, turn: u32, faction: u8) -> *mut c_char`
       - Call dialogue_state.get_pending(trigger, turn, faction)
       - Serialize matching entries as JSON array of objects: [{"id":"...", "text":"..."}]
       - Return JSON string (caller frees). Return "[]" if no matches or no dialogue loaded.

    4. Add `use crate::dialogue::DialogueState;` to ffi.rs imports.

    5. Create norrust_core/tests/dialogue.rs integration test:
       - test_dialogue_ffi_round_trip: create engine, load data, load crossing_dialogue.toml, query turn_start at turn 1 (should get scenario_start with turn=None or a turn 1 entry), verify JSON contains text
       - test_dialogue_ffi_one_shot: query same trigger twice, second returns "[]"
  </action>
